# File Lancet 技术路线纲要

## 一、核心技术栈选型

基于项目对文件处理性能及架构扩展性的要求，确定以下技术组合：

### 开发语言
- **C# (.NET 8 / .NET 9)**
  - 理由：利用 .NET 强大的原生库（System.IO.Compression, System.Xml.Linq）处理文件底层结构；强类型系统确保数据模型的严谨性；跨平台编译能力成熟。

### UI 框架
- **Avalonia UI**
  - 理由：相比 .NET MAUI，Avalonia 在桌面端（尤其是 Linux）拥有更稳定的渲染一致性和更成熟的控件生态（如 TreeDataGrid）；其基于 Skia 的自绘引擎保证了工具界面的像素级还原。

### 架构模式
- **MVVM**
  - 理由：实现 UI 与业务逻辑的彻底解耦，便于单元测试核心解析功能。

### 开发工具
- Visual Studio 2026

## 二、系统架构设计

为了满足"逐步增加可拆解文件类型"的扩展性需求，系统采用分层架构与策略模式相结合的设计。

### 分层架构
- **表现层**：负责界面展示与用户交互（Avalonia Views）。
- **视图模型层**：负责状态管理与数据绑定（ViewModels）。
- **业务逻辑层**：核心解析引擎，负责调度不同的文件解析器（Services）。
- **数据模型层**：定义通用的文件节点结构与解析结果标准（Models）。

### 插件化解析策略
- **接口定义**：定义通用解析接口（如 `IFileLancetParser`），规定 `CanParse`（判断是否支持）和 `Parse`（执行解析）的标准行为。
- **适配器实现**：针对每种文件格式（EPUB, PDF等）实现独立的解析器类。
- **工厂分发**：核心引擎根据文件头或扩展名，自动匹配并实例化对应的解析器。

## 三、关键开发思路

### 数据驱动 UI
- 界面不直接操作文件，而是操作内存中的标准数据模型（FileNode树）。
- 解析器将物理文件转换为标准 JSON/对象结构，UI 仅负责渲染这些对象。

### 懒加载机制
- 针对大文件（如数百兆的 EPUB），在构建树形结构时不立即读取文件内容。
- 仅在用户点击节点或请求预览时，才从磁盘/压缩包中读取具体字节流，以优化内存占用。

## 四、开发阶段规划

项目将分为四个阶段逐步推进，确保每个阶段都有可运行的成果。

### 阶段一：核心地基与 EPUB 解析引擎

本阶段的核心目标是构建一个不依赖 UI 的控制台级解析内核，确保数据提取的准确性与架构的通用性。

**目标 1：构建通用数据模型**

- **任务 1.1：定义文件节点模型**
  - 设计 `FileNode` 类，包含 `Name`（名称）、`Path`（路径）、`Type`（枚举类型：文件夹/HTML/CSS/图片等）、`Children`（子节点列表）以及 `Description`（用于状态栏的描述）。
  - 设计 `NodeType` 枚举，覆盖 EPUB 涉及的所有主要文件类型。

- **任务 1.2：定义解析结果模型**
  - 设计 `FileDetails` 类，用于承载中栏显示的详细数据。包含物理属性（大小、MIME 类型）和逻辑属性（书名、作者、EPUB 版本、出版社）。
  - 确保模型具有空值处理能力，以适应不同文件类型的属性差异。

**目标 2：实现 EPUB 解析策略**

- **任务 2.1：搭建解析器接口**
  - 定义 `IFileLancetParser` 接口，规定 `Parse(string filePath)` 方法。
  - 建立解析器工厂的雏形，用于后续根据文件扩展名分发任务。

- **任务 2.2：实现 EPUB 核心逻辑**
  - **容器解包**：利用 `System.IO.Compression` 读取 ZIP 流，定位 `META-INF/container.xml`，获取 `.opf` 文件路径。
  - **OPF 解析**：利用 `System.Xml.Linq` 解析 OPF 文件。提取节点获取书名、作者；提取构建资源列表；提取确定阅读顺序。
  - **目录构建**：解析 `toc.ncx` 或 `nav.xhtml`，构建层级化的目录树结构。
  - **内容懒加载**：在构建树时仅存储 ZIP Entry 引用，不立即读取字节流，确保大文件加载速度。

**里程碑交付物**
- 一个控制台应用程序，输入 EPUB 路径，能在控制台打印出结构化的 JSON 数据，包含完整的书籍元数据和文件树。

### 阶段二：三栏式界面实现

本阶段重点在于利用 Avalonia UI 构建交互界面，并将第一阶段的数据模型通过 MVVM 模式绑定到视图上。

**目标 1：搭建 MVVM 框架**

- **任务 1.1：配置依赖注入**
  - 设置 `ViewModelLocator` 或配置服务容器，确保 ViewModel 能获取到解析器实例。

- **任务 1.2：实现主窗口 ViewModel**
  - 创建 `MainViewModel`，包含 `FileTreeNodes`（左栏数据源）、`SelectedNode`（当前选中节点）、`NodeDetails`（中栏数据源）等属性。
  - 实现 `INotifyPropertyChanged` 接口，确保数据变动能实时更新 UI。

**目标 2：实现界面布局与绑定**

- **任务 2.1：构建三栏布局**
  - 使用 `Grid` 控件划分左（250px）、中（*）、右（400px）三列，并添加 `GridSplitter` 实现拖拽调整宽度。
  - 底部添加状态栏区域。

- **任务 2.2：左栏-文件树视图**
  - 使用 `TreeView` 控件绑定 `FileTreeNodes`。
  - 实现 `DataTemplate`，根据节点类型（文件夹/文件）显示不同的图标。
  - 绑定 `SelectedItem` 事件，当用户点击树节点时，更新 ViewModel 中的 `SelectedNode`。

- **任务 2.3：中栏-动态属性视图**
  - 设计属性面板布局：顶部显示文件名和图标，中间使用 `DataGrid` 或 `StackPanel` 列表显示键值对（如"作者：XXX"，"大小：XXX"）。
  - 实现数据触发器：当选中 OPF 节点时，高亮显示书籍元数据区域；当选中普通文件时，显示物理属性区域。

- **任务 2.4：底部-状态栏联动**
  - 绑定状态栏文本属性，根据 `SelectedNode.Description` 动态显示"这是定义书籍阅读顺序的配置节点"等提示语。

**里程碑交付物**
- 一个可运行的桌面应用，能加载 EPUB 文件，左侧展示树状图，点击节点中间显示详细属性，但右侧预览区暂时为空或仅显示"待开发"。

### 阶段三：内容预览与交互完善

本阶段致力于提升用户体验，实现"所见即所得"的预览功能，并完善文件输入的便捷性。

**目标 1：实现右栏预览功能**

- **任务 1.1：文本与代码预览**
  - 对于 CSS、XML、HTML 源码，集成代码高亮控件（如 AvaloniaEdit），提供语法高亮显示。

- **任务 1.2：HTML 渲染预览**
  - 调研并集成 Avalonia 的 WebView 控件（如 Avalonia.Web 或 WebView 控件），用于渲染 XHTML 内容。
  - 解决资源加载问题：确保 HTML 中的相对路径图片（如）能正确从 ZIP 流中读取并显示。

- **任务 1.3：图片预览**
  - 对于 JPEG/PNG 节点，使用 `Image` 控件直接显示二进制流。

**目标 2：完善文件输入交互**

- **任务 2.1：拖拽加载**
  - 在主窗口启用 `AllowDrop=true`。
  - 处理 `Drop` 事件，提取文件路径并调用解析器。

- **任务 2.2：文件对话框与关联**
  - 添加顶部菜单栏"文件->打开"，调用系统文件选择器。
  - 配置 `.epub` 文件关联，支持双击文件直接启动应用并加载。

**目标 3：异常处理与反馈**

- **任务 3.1：加载状态管理**
  - 在解析大文件时，UI 显示"正在解析..."的遮罩层或进度条，防止界面假死。

- **任务 3.2：错误捕获**
  - 捕获 `ZipException` 或 `XmlException`，在状态栏或弹窗中友好提示"文件损坏"或"非标准 EPUB"。

**里程碑交付物**
- File Lancet v0.1 Beta 版。具备完整的三栏分析能力，支持拖拽打开，右侧能预览正文和图片。

### 阶段四：扩展性验证与优化

本阶段旨在验证架构的灵活性，并为未来的多格式支持打下基础。

**目标 1：架构重构与插件化**

- **任务 1.1：提取公共基类**
  - 将 EPUB 解析器中的通用逻辑（如文件流读取、异常处理）提取到 `BaseParser` 基类中。

- **任务 1.2：完善接口定义**
  - 确保 `IFileLancetParser` 接口足够通用，能容纳 PDF 或其他格式的特殊属性。

**目标 2：扩展性验证（试点）**

- **任务 2.1：实现"虚拟"第二格式**
  - 开发一个简单的 `PlainTextParser`（支持 .txt）或 `JsonParser`。
  - 验证在加载 txt 文件时，应用能否自动切换解析策略，并在界面正确显示 txt 的文件结构和内容。
  - 目的：证明架构不需要修改核心代码即可支持新格式。

**目标 3：性能优化与发布**

- **任务 3.1：内存优化**
  - 检查大文件加载时的内存占用，确保 ZIP 流在使用后及时释放。

- **任务 3.2：跨平台打包**
  - 编写发布脚本，生成 Windows（.exe/.msi）和 Linux（AppImage/.deb）安装包。

**里程碑交付物**
- 具备多格式支持能力的架构框架，以及 File Lancet v0.1 稳定版。

## 五、测试要求

### 单元测试

- **测试框架**：使用 xUnit 或 NUnit 作为测试框架，配合 FluentAssertions 进行断言。
- **覆盖率目标**：核心业务逻辑层（解析器、数据模型）的代码覆盖率不低于 80%。
- **测试范围**：
  - 解析器接口实现（`IFileLancetParser` 的各个实现类）
  - 数据模型验证（`FileNode`、`FileDetails` 等属性的正确性）
  - 工具类方法（文件路径处理、MIME 类型识别等）
- **Mock 策略**：使用 Moq 或 NSubstitute 模拟文件系统和外部依赖，确保测试的独立性。

### 集成测试

- **测试场景**：
  - 完整 EPUB 文件解析流程（从文件加载到数据模型生成）
  - 多种 EPUB 版本兼容性测试（2.0、3.0、3.1）
  - 异常文件处理（损坏的 ZIP、缺失关键文件、非标准格式）
- **测试数据**：准备多种类型的测试 EPUB 文件（简单文本、复杂排版、多图片、大文件）
- **自动化**：集成到 CI/CD 流程，每次提交自动运行集成测试。

### UI 测试

- **测试工具**：使用 Avalonia UI 测试框架或 Appium 进行端到端测试。
- **测试重点**：
  - 三栏布局的响应式行为（窗口大小变化、拖拽调整宽度）
  - 文件树节点的展开/折叠、选中状态
  - 拖拽文件打开功能
  - 预览区域的正确渲染
- **视觉回归**：关键界面截图对比，防止 UI 意外变更。

### 性能测试

- **测试指标**：
  - 大文件（>100MB）加载时间不超过 5 秒
  - 内存占用不超过文件大小的 3 倍
  - 树形结构渲染 10000+ 节点时保持流畅
- **测试方法**：使用 BenchmarkDotNet 进行基准测试，记录关键操作的性能数据。
- **压力测试**：连续打开/关闭多个大文件，检测内存泄漏和资源释放情况。

### 跨平台测试

- **测试环境**：
  - Windows 10/11（x64、ARM64）
  - Ubuntu 22.04 LTS、Fedora 39
- **测试内容**：
  - 文件路径处理（Windows 反斜杠 vs Linux 正斜杠）
  - 界面渲染一致性
  - 打包后的安装包可正常安装运行
- **兼容性**：测试不同 .NET 运行时版本（.NET 8、.NET 9）的兼容性。

## 六、质量与进度要求

- **代码规范**：严格遵循 C# 命名规范（PascalCase/camelCase），保持代码整洁。
- **异步编程**：文件 I/O 操作必须使用异步方法（async/await），严禁阻塞 UI 线程。
- **进度同步**：每完成一个阶段，需进行功能演示与代码走查，确保方向无误。
